3. Heuristic Mathematics & Feature Extraction

Machine learning models cannot read raw strings; they require strict mathematical representations of data. PhishGuard's model_inference.py uses several custom heuristic algorithms to convert a URL into a 22-dimensional feature vector.

Specific Heuristic Algorithms and Their Mathematics

1. Shannon Entropy (Randomness Detection)

Malware often uses Domain Generation Algorithms (DGAs) to create random URLs like http://x8k-9z2p-ql.com to avoid blacklists. Entropy mathematically proves this randomness.

Formula: H(X) = - Σ [ P(x_i) * log₂(P(x_i)) ]

Implementation: For every unique character in the URL, the algorithm calculates its probability P(x) = count(char) / total_length. It sums the negative log base 2 of these probabilities. The higher the number, the more random (and suspicious) the URL.

2. IP Address Obfuscation Detection

Legitimate sites use DNS (e.g., google.com). Phishers often bypass DNS completely by using raw IP addresses or hex-encoded IPs.

Algorithm: Regex Matching ^\d{1,3}(\.\d{1,3}){3}$
Output: Returns 1 (True) if the hostname is a direct IPv4 address, immediately increasing the threat score.

3. TLD Mimicry in Subdomain

A classic phishing attack is to put a trusted brand in the subdomain (e.g., paypal.com.secure-update.xyz). The user sees "paypal.com" and trusts it.

Algorithm Logic:
1. Use tldextract to separate the URL into Subdomain, Domain, and Suffix/TLD.
2. Evaluate: IF Suffix IN Subdomain THEN return 1
3. In paypal.com.secure-update.xyz, the suffix is xyz, but the subdomain contains com, triggering a massive red flag.

4. Digit to Character Ratio Mapping

Ratio_Digits_URL Formula:
R = Σ(is_digit(char)) / Total_URL_Length

Legitimate domains rarely consist mostly of numbers. A high ratio mathematically penalizes the URL.

5. Phish Hints Keyword Density

The system counts occurrences of high-risk keywords (secure, login, update, verify, banking) inside the raw string to create a density score.

4. Machine Learning Deep Dive: XGBoost & MLP

Model 1: XGBoost (Extreme Gradient Boosting) - The Decision Trees

XGBoost is not just a single model; it is an "Ensemble" of Decision Trees.

🌳 How Decision Trees Work (Real Usage):
Imagine a flowchart.
- Root Node (Top): The model asks, "Is the URL length > 54 characters?" If Yes, go left. If No, go right.
- Internal Node: "Does the URL contain an IP address?"
- Leaf (Bottom): The final decision, e.g., "90% chance it is Phishing."

A single tree is often inaccurate. Gradient Boosting fixes this. We build 150 trees sequentially.
Tree 1 predicts the data. It gets 10 URLs wrong.
Tree 2 is built specifically to fix Tree 1's mistakes.
Tree 3 fixes Tree 2's mistakes.
By the time we hit Tree 150, the combined "forest" is incredibly precise.
The Mathematics of the Split:
XGBoost decides exactly where to split a node (e.g., why >54 chars instead of >60 chars?) by calculating the Similarity Score and Gain. It uses Taylor Expansion (calculus derivatives) to find the exact split point that minimizes the Log-Loss the fastest.

Model 2: Multi-Layer Perceptron (MLP) - The Deep Neural Network

While trees use hard "Yes/No" splits, an MLP Neural Network mimics the human brain, allowing it to find deeply hidden, non-linear correlations (e.g., "A long URL is fine, UNLESS it also has 3 hyphens AND no SSL").

🧠 How the MLP Neurons Work (Real Usage):

1. Input Layer (22 Neurons): Each of our 22 heuristic features (Entropy, Length, IP flag) enters one neuron.
2. Hidden Layer 1 (128 Neurons): Each of the 128 neurons receives data from ALL 22 inputs. But they apply Weights. A neuron acting as a "Security Expert" might multiply the "IP flag" by a massive weight (importance), but ignore the "URL Length".
3. Hidden Layer 2 (64 Neurons): These neurons look at the conclusions from the first 128 and synthesize them into higher-level thoughts.
4. Output Layer (2 Neurons): Returns the final probability of [Safe, Phishing].
The Mathematics of the MLP:
1. Activation (ReLU): f(x) = max(0, x). If a neuron's calculation is negative, it outputs 0. This stops irrelevant data from propagating.
2. Backpropagation: When the MLP makes a wrong guess during training, it calculates the Cross-Entropy Loss. It then uses calculus (Chain Rule) to send that error backwards through the network, slightly adjusting every single "Weight" so it makes a better guess next time.

5. Comprehensive Flow of Data at Each Layer

To fully understand PhishGuard, one must trace the flow of data from the exact moment a user clicks a malicious link to the moment the block is confirmed on the Ethereum network.

Layer 1: The Client/Edge Layer (Data Acquisition)

1. The Chrome Extension's manifest.json triggers the background.js service worker using the chrome.webNavigation.onCommitted API.
2. content.js extracts the raw URL string and captures a Base64 encoded screenshot of the viewport using the chrome.tabs.captureVisibleTab API.
3. This payload is POSTed to the backend API at /api/v1/scan.

Layer 2: The Gateway & Heuristics Layer (Data Enrichment)

1. The FastAPI server receives the payload.
2. It triggers multiple Parallel Coroutines using Python's asyncio.gather().
3. Coroutine A (WHOIS): Queries Domain Age.
4. Coroutine B (SSL): Checks for HTTPS certificates.
5. The original JSON is now enriched with contextual intelligence before AI processing.

Layer 3 & 4: AI & Vision Inference

1. The 22 features are mapped to a NumPy array, scaled, and fed into the XGBoost + MLP Ensemble, outputting a threat probability (e.g., 87.5%).
2. Concurrently, the Vision Engine decodes the Base64 image into an OpenCV multidimensional matrix. It converts it to Grayscale and uses skimage.metrics.ssim to compare its structure against known banking logins.

Layer 5: The Web3 Blockchain Layer (Immutable Logging)

1. The incident is written to the local PostgreSQL database for immediate rendering on the React SOC Dashboard.
2. IPFS Pinning: The massive JSON report (including the ML probabilities and base64 evidence) is pushed to the InterPlanetary File System (IPFS), returning a CID (Hash).
3. Blockchain Transaction: The backend uses an Ethereum Private Key to sign a transaction, calling the logThreat(url, cid) function on the Sepolia Smart Contract. The data is now immutable.

6. Authentication & Zero-Trust Architecture (RBAC)

A core innovation of PhishGuard is the elimination of vulnerable standalone login pages for the Security Operations Center (SOC) dashboard. We employ a Hardware-Linked Zero-Trust methodology.

  • Extension as an Auth Key: Instead of logging into a website, analysts register and authenticate directly through the Chrome Extension popup. The extension requests a JSON Web Token (JWT) from the FastAPI backend.
  • Encrypted Storage: The JWT is securely stored in chrome.storage.local. It is completely isolated from web scripts, mitigating Cross-Site Scripting (XSS) theft.
  • URL Parameter Injection: When an analyst clicks "Open Dashboard", the extension securely injects the JWT into the URL payload. The React frontend consumes this, verifying the signature cryptographically before rendering the UI.
  • Role-Based Access Control (RBAC): The JWT payload contains the user's role.
    • Standard User: Can only view their locally blocked threats.
    • Security Analyst: Can view the global threat matrix but cannot trigger model retraining.
    • Administrator: Has root-level access, can view all Ethereum transaction hashes, and orchestrate ML retraining pipelines.

7. Web3 Blockchain & Gas Optimization

Storing data on the Ethereum blockchain is incredibly expensive ("Gas Fees"). Storing an entire JSON threat report (which contains a base64 image) directly in a Smart Contract would cost hundreds of dollars per transaction.

🎓 Explaining to Invigilators (Gas Optimization):
To solve the exorbitant costs of blockchain storage, PhishGuard uses a hybrid approach with IPFS (InterPlanetary File System).

1. The massive 5MB JSON threat report is uploaded to the decentralized IPFS network.
2. IPFS returns a tiny 46-character string called a CID (Content Identifier). The CID is a cryptographic hash of the exact file contents.
3. We write ONLY the malicious URL and the 46-character CID to the Ethereum Smart Contract. This reduces the Ethereum transaction cost by 99.9%, while maintaining 100% cryptographic proof that the data has not been tampered with.

8. Deployment Architecture (Docker & CI/CD)

PhishGuard is designed to be easily deployable in modern cloud environments, proving industry-readiness beyond just local execution.

  • Containerization (Docker): The entire backend, frontend, and PostgreSQL database are containerized using docker-compose.yml. This ensures the application runs identically on Windows, Linux, and macOS without dependency conflicts.
  • FastAPI on Uvicorn: The backend uses Uvicorn, an ASGI (Asynchronous Server Gateway Interface) server. This allows Python to handle thousands of concurrent API requests from browser extensions without locking the main thread.
  • Frontend via Vite & Nginx: The React SOC Dashboard is bundled using Vite for rapid compilation and served in production via a reverse-proxy (Nginx), ensuring high availability and static file caching.

9. Future Roadmap & Ethical Considerations

Future Roadmap (MLOps & Retraining)

The next architectural phase for PhishGuard is implementing a dynamic MLOps Retraining Pipeline. Currently, models are static. The future pipeline will use Apache Airflow or Celery to aggregate false-negatives (phishing sites missed by the ML but caught by a human) at the end of each week, automatically retraining the Random Forest and MLP ensemble, and hot-swapping the .pkl file in production with zero downtime.

Ethical Considerations & Privacy

As a tool that operates in the browser and captures DOM snapshots, user privacy is paramount.

  • Selective Capture: The Chrome Extension only captures screenshots and DOM data when an initial heuristic flag is triggered (e.g., suspicious URL). It does not record session data on known safe sites like google.com.
  • No PII Storage: The FastAPI backend strips out password fields, input boxes, and Personally Identifiable Information (PII) before calculating the SSIM score or pinning the report to IPFS. The blockchain only retains the URL string and the abstracted CID hash.
